# 🚧 Future Work

## 🎯 模糊测试扩展方向

---

### 🔧 1. Library-Level API Driver 自动生成

**目标**
为下载下来的库中的每个可用 API 自动生成测试驱动（driver），使其能够被调用，并生成用于模糊测试的输入用例。

**方法**
在分析 `library/` 文件夹下的所有可利用源码文件时，除了收集 API 的函数签名和实现外，还应挖掘以下信息以辅助 LLM 自动生成调用驱动：

* ✅ **函数注释**：理解参数语义与返回值含义
* 📄 **文档描述**：补充 API 的使用规范与调用约束
* 🔁 **相似函数的调用方式**：例如，在处理 `compress2()` 时，同时参考 `compress()` 的调用案例，有助于构造合理的参数与调用序列

---

### 🔍 2. 高风险 API 驱动与模糊测试优先级排序

**目标**
识别库中更可能引发缺陷的“高风险” API，并优先为其生成 fuzzing driver，从而在更少时间内发现更多漏洞。

**方法**

* 🔬 **静态分析驱动**：对 API 实现代码进行静态分析，检测如内存操作、边界访问、类型转换等敏感逻辑，识别潜在高风险代码块
* 🤖 **基于 LLM 的模式识别**：总结以往 fuzzing 过程中发现的漏洞模式（如 use-after-free、heap overflow），并让 LLM 扫描新 API 实现以发现可能复现此类模式的代码位置

**预期效果**
相比于对 API 进行无差别随机模糊测试，该方法应能以显著更高的效率发现真实缺陷。
